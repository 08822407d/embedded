# Atom_Roller_Bala — 项目接手入口

> ⚠️ **如果你是新的 Claude Code 会话:你没有任何对话记忆。**
> 本项目的事实只存在于 `docs/` 目录和 git 历史里,不在你的"印象"中。
> **第一步:打开并读 [`docs/STATUS.md`](docs/STATUS.md)** —— 它告诉你现在进行到哪、下一步做什么、卡在哪。

## 交互方式:语音听写输入(重要)
本项目的对话**主要由语音听写产生**:用户说话 → 语音转文字(STT)→ 文本发给你,因此你收到的文字**可能带有识别误差**(同音字、词边界错位、标点缺失、专业术语被听错等)。

- **按意图理解,别死抠字面**:遇到读不通或明显不对的词,结合上下文与本项目领域(嵌入式开发、需求讨论、数学 / 物理模型)推断用户真正想说的,心里纠正后再回应。
- **领域聚焦、话题不发散**,绝大多数误识别都能靠上下文确定,纠错难度不大 —— 大胆按最合理的技术含义解读。
- **少打断**:只有当某处误识别会影响关键决策、且上下文无法确定时,才用一句话简短确认;不要逐字逐句反复追问。

## 工作原则:工程严谨(必守)
本项目是**自动控制入门性质的工程项目**,严谨第一。
- **尊重数学与计算机的事实**:不得为迎合 / 讨好用户而编造不符合科学理论或事实的回答。不确定就说不确定并给出验证办法,**绝不自信地胡说**;算不出 / 没把握的,讲清依据与边界,不谎报成功。
- **可行性已有保障**:同类系统(反作用轮倒立摆自平衡)他人已成功实现(见 [`docs/FACTS.md`](docs/FACTS.md) F‑9)→ 目标**确定可达**。故遇失败默认是**实现 / 参数 / 接线 / 标定**问题,回到 FACTS + 系统排查;**不要归因于"原理上不可能",也不要轻易放弃**。可行 ≠ 可以谎报成功 —— 二者由上一条约束。

## ⛔ 操作红线(安全 · 必守)
**未经用户在本会话中明确指示,不得:① 开始编码实现、② 烧录(flash / `pio ... upload`)、③ 驱动 RollerCAN / 让电机运转。**
- 原因:开发板常**无人值守**;一旦机体翻转到不可恢复的姿态,用户不在场,无法手动扶正。
- 默认只做:需求 / 设计 / 架构 / 分析 / 文档(本会话主业),无需额外许可。
- 烧录与"让电机动"是有物理后果的操作:**必须用户在场且明确要求**才做。`pio ... upload` 在 `.claude/settings.json` 已设为每次必问(`ask`)作兜底,但真正的红线是上面这条 —— **别主动发起**。

## 本项目怎么运作
- **Claude Code(本会话,主)**:需求分析、方案与架构设计、任务拆解、验收 → 写进 `docs/`。
- **Codex**:照 `docs/DESIGN.md` 写代码、跑测试、调试 → 产出在 `src/`。
- **真相在磁盘 + git;对话随时会丢,所以一切结论都要落盘。**

## 维护规则(必守,展开见 HANDOFF)
1. **收尾必更 [`docs/STATUS.md`](docs/STATUS.md),然后 `git commit`** —— 上下文随时会断,不提交 = 没存。
2. 做了非显然的技术选择 → 追加一条到 [`docs/DECISIONS.md`](docs/DECISIONS.md)。
3. 需求 / 设计变化 → 同步改 [`docs/REQUIREMENTS.md`](docs/REQUIREMENTS.md) / [`docs/DESIGN.md`](docs/DESIGN.md)。
4. 新建代码模块时,可在该模块目录放一个**局部 `CLAUDE.md`** 记录局部约定 —— CLAUDE.md 是树状的,按需就近生长,根文件保持轻量。
5. 核实了某事实(实测 / 用户确认 / 手册)→ 记入 [`docs/FACTS.md`](docs/FACTS.md);**调试反复失败时先读它,别怀疑已验证事实**(完整规则见该文件头)。

> 文档定位:`docs/` 主要是给 **AI agent 当设计 / 分析 / 实验的事实与依据**用的 —— 按"AI 好检索、精确、不被误读"来写,不为人工阅读牺牲准确或加修饰。

完整的接手与恢复流程见 **[`docs/HANDOFF.md`](docs/HANDOFF.md)**。
