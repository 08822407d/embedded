# DECISIONS — 决策记录

## D-0001：使用短 AGENTS.md + context 文件，而不是把所有资料塞进 AGENTS.md

原因：

- Codex 项目说明存在默认大小限制。
- 大量文档塞进 AGENTS.md 容易迁移困难、重复、过期。
- context 文件可版本化、可审计、可恢复。

结果：

- `AGENTS.md` 只保留硬规则。
- 设备事实、计划、交接、文档摘录分别放入 `context/`。

## D-0002：新增 CONFIG_REGISTRY 作为配置检测、依赖和方案取舍台账

原因：

- 仅记录已采纳配置不足以支持新设备/新会话判断“是否已经配置好”。
- 同一问题可能有多种方案；实际采纳方案可能是阶段性折中，不一定长期最佳。
- 若不集中记录检测命令、依赖、替代方案和取舍原因，后续任务容易重复联网检索、重复推理，并增加 token 成本。
- 未记录的现场配置可能来自用户手工改动或官方镜像默认，不能直接覆盖。

结果：

- 新增 `context/CONFIG_REGISTRY.md`。
- 每个配置类任务先查配置项，运行登记的检测命令，再判断 `compliant` / `missing` / `drift` / `unrecorded` / `unknown` / `deferred`。
- 缺失配置必须按依赖关系和正确配置步骤推进；中高风险动作仍需用户确认。
- 未记录配置先登记证据和影响范围，不直接覆盖。
- 替代方案、当前采纳方案和阶段性整体最佳方案集中维护在 `CONFIG_REGISTRY.md`。
