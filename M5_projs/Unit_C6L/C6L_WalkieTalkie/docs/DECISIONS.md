## #001 — 2026-06-21 — LoRa 测试:有界运行+自动关射频、等串口再发
- **决定**:LoRa 测试固件**有界运行**(PINGER 发满 N=30 包 / PONGER 空闲 20s 超时)后 `radio.sleep()` 关射频;**等 USB-CDC 串口打开(带 30s 超时)再开始打印与收发**;主机抓取脚本先挂载后复位、对齐 t=0。
- **为什么**:(a)兜底——不让设备测试后持续对外发射射频(呼应红线);(b)原生 USB-CDC 一次性 boot 日志竞态(F‑13),不对齐就漏 banner / 早期包,等串口 + 先挂载后复位双保险确保抓全。
- **否决**:无限循环发射(违红线、且无明确"测试结束"信号)。

## #002 — 2026-06-21 — C6L 定位为纯 LoRa 数据 modem(非主控)
- **决定**:C6L **只作 SX1262 LoRa 的数据传输 modem**——数据内容透明(语音/文字/图片/任意字节),不做主控、不做编解码;数据来源与编解码/运算放在**外接主机**。优先**可靠正确**,**不要求实时**。
- **为什么**:ESP32-C6 算力不足以承担语音等编解码;用户要的是"可靠、非实时、任意数据"的无线透传管道。
- **影响**:固件 = 串口帧(主机↔C6L)⇄ LoRa 分片收发 + ARQ(序号/CRC/ACK/重传);C6L 上不做音频/编解码。射频路线已定 = LoRa(SX1262,板载,F-15~F-21)。
- **待选(未决)**:路线 A 借用 Reticulum/RNode(需移植到 C6L)vs 路线 B 自研串口+ARQ(基于已验证 RadioLib)。详见 REFERENCE_lora_transport.md §2。

## #003 — 2026-06-24 — modem 路线 / 主机接口 / 射频参数定调(用户答复 5 点)
- **路线 = A(Reticulum + RNode)**:协议栈(可靠/分片/加密/寻址)用现成的;自研只剩"把 RNode 固件**移植到 C6L**"(主要工作量,但有界:引脚/初始化已知 F-16/F-17/F-21,难点是 RF 控制线走 I2C 扩展器、需在 RNode 里加 C6L 板支持)。主机侧装 Reticulum(Python)。
- **外接主机 = 当前开发机(PC / Ubuntu)先行**;长期可能换成板载麦克风+喇叭的 **ESP32-S3**。
- **主机↔C6L 接口 = 串口(UART 类),不用 I2C**:
  - PC 主机 → **USB-CDC**(PC 无 Grove 口,C6L 的 USB-C 直连,已在用);未来 ESP32-S3 主机 → **Grove UART(G4/G5)**。
  - 为何不纠结 UART vs I2C 带宽:① 路线 A 的 RNode/Reticulum 本就是**串口(KISS)**接口,**不支持 I2C**;② **接口带宽根本不是瓶颈**——UART(ESP32-C6 spec 可达 ~5Mbps,实用 921600~2Mbps)、I2C(FM+ 1MHz,有效 <100KB/s)都**远高于 LoRa 的 ~0.3~20kbps**,选谁对吞吐无差别 → 按兼容性选 UART。
- **目标距离 = 假定 1km**:LoRa 轻松覆盖(1km 偏短)→ **不用 SF12**,起步 **SF8/BW125(或 SF7)**,实测不稳再升 SF;比测试用 SF12 吞吐大幅提升。
- **频段 = 868~923MHz(SX1262 高频),默认 868.0MHz**(与实测/天线一致);两端必须同频;region 合规由用户按所在地确认。
- **占空比(用户让我设计)= 10% 长期 airtime 自设上限**(个人点对点,平衡吞吐与信道礼貌/散热);**注**:EU868 法定为 1%——要严格合规则设 1%(但 1% 会让图片传输极慢);RNode 固件自带 airtime-limit,届时按此配置。
- **载荷/耗时**:用户暂无定数 → 工作假设:文字(秒级)、短语音消息(Codec2 ~1-3kbps,数秒~分钟级)、小图(几十KB,分钟级);后续按真实用例细化。
