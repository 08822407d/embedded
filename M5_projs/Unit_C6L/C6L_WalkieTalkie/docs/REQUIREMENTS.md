# REQUIREMENTS — 需求(做什么 / 为什么 / 怎样算完成)

> 规则:已确认的写成肯定句并标日期;未确认的放「待确认」区,别当既定事实用。

## 1. 目标(一句话)
一对 M5Stack Unit-C6L,做对讲机/通信终端的**无线传输层**。**C6L 仅作 SX1262 LoRa 的数据 modem(调制解调器)**,不做主控、不做编解码。(2026-06-21 用户确认)

## 2. 架构定位(关键,2026-06-21 确认)
```
[外接主机A(大脑)] --串口--> [C6L-A: LoRa modem] ))) LoRa (((  [C6L-B: LoRa modem] --串口--> [外接主机B(大脑)]
```
- **C6L 只负责无线收发**,对数据内容**透明/不关心**:可能是语音、文字、图片、任意字节流。
- **数据来源 + 编解码 + 运算全在外接主机**(ESP32-C6 算力不足以承担语音等编解码,故不作主控)。
- 外接主机经**串口**(USB-CDC 或 Grove UART,待定)把字节交给 C6L,C6L 用 LoRa 送到对端 C6L,再交给对端主机。

## 3. 功能需求
- [ ] **数据透明传输**:不解释内容,按字节流可靠送达对端(语音/文字/图片/任意)。
- [ ] **分片 + 重组**:载荷可能远大于 LoRa 单包上限(SX1262 ≤255B)→ 必须分片再重组。
- [ ] **可靠传输(ARQ)**:序号 + CRC + ACK + 超时重传,保证**正确、无丢、有序**。
- [ ] **主机↔C6L 串口帧接口**:带帧定界(如 KISS),把主机字节流切成可发的块。
- [ ] (可选)流控/背压、链路状态上报(RSSI/SNR/进度)给主机。

## 4. 非功能需求
- **不要求实时性**(用户明确)。带宽极小(LoRa,几百 bps~几 kbps)可接受。
- **首要指标 = 正确性/可靠性**:数据必须完整无误到达。
- 距离、续航、功率:**待定**(距离要求决定 SF/BW,从而决定吞吐——见 [`REFERENCE_lora_transport.md`](REFERENCE_lora_transport.md))。

## 5. 验收标准(MVP)
- 任意字节流(含较大文件/一张小图)从主机A → C6L-A → LoRa → C6L-B → 主机B,**端到端逐字节正确**(校验一致),允许慢、允许重传,不允许错/缺。

## 已定(2026-06-24,详见 DECISIONS #003)
1. **主机/接口**:先用**当前 PC(Ubuntu)**作主机,**USB-CDC** 连 C6L(未来换 ESP32-S3 则走 Grove UART);**不用 I2C**(RNode 走串口,且接口带宽远非瓶颈)。
2. **路线 = A(Reticulum + RNode)**;主要工作 = 把 RNode 固件移植到 C6L。
3. **距离假定 1km** → 起步 **SF8/BW125(或 SF7)**,非 SF12。
5. **频段 868.0MHz**;**占空比自设 10%**(EU 法定 1%,要严格合规再降)。

## 待确认(剩余)
4. **典型载荷与可接受耗时**:用户暂无定数。工作假设:文字秒级 / 短语音消息分钟级 / 小图(几十KB)分钟级——待真实用例细化。
- region 合规(所在地是否允许 868~923);1km 实测后 SF/BW 标定。
